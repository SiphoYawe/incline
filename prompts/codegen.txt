{need_spec}

ARCHETYPE (the only kind of tool you may build): {archetype}

=== OUTPUT FORMAT (read carefully) ===
- Output EXACTLY ONE complete HTML document and NOTHING else.
- Start your response with "<!DOCTYPE html>" and end with "</html>".
- NO prose, NO explanation, NO markdown, NO ``` code fences. Just the raw HTML.

=== HARD GENERATION CONTRACT (every item is mandatory) ===
1. ONE self-contained HTML file. All CSS in a single inline <style> and all JS in
   a single inline <script>. NO external requests of any kind: no <link>, no CDN,
   no fonts/images/scripts fetched from the network, no frameworks, no build step.
   It must work when opened directly from disk with no internet. Keep it under 60 KB.

2. The tool takes the user's input IN THE BROWSER (a <textarea> and/or a
   <input type="file"> for text tools; <input> fields for a calculator) and, when
   they click the primary action button, produces a VISIBLE result computed in JS.
   Parse and transform their actual input — show specifics drawn from it (counts,
   detected values), never a generic placeholder.

3. TWO result states:
   a) A FREE PREVIEW / teaser that is always visible after they run the tool — show
      enough of the real answer (e.g. the first group, or the headline number) that
      they believe it works on their data.
   b) The FULL result lives inside an element with exactly id="locked-result", which
      is hidden by default (e.g. the element has the `hidden` attribute / display:none).
      On page load, read new URLSearchParams(location.search).get("unlocked"). If it
      equals the string "1", REVEAL #locked-result and show a short thank-you. Do this
      automatically on load — no reload, no extra click.

4. A pay-to-unlock button labelled "Pay to unlock your result" (or
   "Unlock & download · PayPal"). Its click handler MUST run exactly this fetch,
   with these EXACT literal placeholder strings (do NOT substitute real values):

   fetch("{{BASE_URL}}/pay/create", {
     method: "POST",
     headers: { "Content-Type": "application/json" },
     body: JSON.stringify({ tool_id: "{{TOOL_ID}}" })
   })
   .then(function(r){ return r.json(); })
   .then(function(d){ window.location.href = d.approval_url; });

   Emit the literal text {{BASE_URL}} and {{TOOL_ID}} verbatim in the HTML — the
   server replaces them before serving. Never hardcode a real URL or id. Parse
   `approval_url` from the JSON response and set window.location.href to it.

5. NO secrets, NO API keys, NO tokens of any kind anywhere in the HTML. There is no
   server logic here except the /pay/create call above. Do NOT invent other endpoints.

6. Persist the user's input across the PayPal redirect: on the pay click, save the
   raw input to localStorage; on load with ?unlocked=1, restore it, recompute, and
   build a client-side download (a Blob) of the full result so the buyer can keep it.
